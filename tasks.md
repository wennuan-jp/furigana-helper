# 片假名外来语溯源功能方案探讨

## 1. 目标
针对片假名（Katakana），在显示振假名（Furigana）的同时或以其他方式（如 Tooltip），提示该单词对应的原始外文单词（如 English, French, German 等）。

## 2. 离线执行 vs API 调用

### 方案 A：离线执行（本地词典）
*   **原理**：在插件中内置一个小型数据库或 JSON 词典，包含常见的片假名及其外文原词。
*   **数据来源**：可以从 [JMDict (EDRDG)](http://www.edrdg.org/jmdict/j_jmdict.html) 中提取带有 `<lsource>` 标签的条目。
*   **优点**：
    *   **隐私性好**：数据不离开用户本地。
    *   **速度极快**：无网络延迟。
    *   **零成本**：无需支付 API 费用。
    *   **离线可用**：在没有网络的情况下也能工作。
*   **缺点**：
    *   **词典体积**：完整的 JMDict 非常大。需要进行精简（只保留片假名和外文原词），即便如此也可能有几 MB。
    *   **维护成本**：需要定期更新内置词典。
    *   **覆盖率有限**：无法涵盖所有的生僻外来语或新造词。

### 方案 B：API 调用（LLM 或 翻译服务）
*   **原理**：将识别出的片假名发送到云端 API（如 Google Translate API, DeepL API, 或 Gemini/OpenAI API），由 AI 返回其原词。
*   **优点**：
    *   **准确度高**：AI 可以结合上下文判断（例如「パン」在不同语境下可能是平底锅还是面包）。
    *   **覆盖率广**：几乎能识别任何外来语。
    *   **体积小**：无需在插件中存储庞大的词典数据。
*   **缺点**：
    *   **网络依赖**：必须联网。
    *   **延迟**：存在网络请求往返时间。
    *   **成本**：大规模调用可能产生费用。
    *   **隐私**：用户划选的内容会发送到第三方服务器。

---

## 3. 具体实现方案建议

建议采用 **「离线优先，按需扩展」** 的策略。

### 核心逻辑步骤：
1.  **形态学分析**：利用现有的 `Kuromoji` 分析选中的文本。
2.  **提取片假名 Token**：筛选出词性为外来语（Katakana）的 Token。
3.  **匹配与查找**：
    *   **Step 1 (离线)**：在插件内置的常用外来语 JSON 词典中查找。
    *   **Step 2 (可选/在线)**：如果离线词典未命中，且用户开启了「增强识别」功能，则调用 API。
4.  **UI 展示**：
    *   方式 1：在 `<ruby>` 标签上方再加一层（嵌套 ruby，但浏览器兼容性较差）。
    *   方式 2：使用 HTML5 `title` 属性或自定义 Tooltip，用户鼠标悬停在片假名上时显示原词。
    *   方式 3：在振假名括号内显示，例如：`パン(pão)`。

## 4. 潜在问题与挑战

1.  **词典体积与性能**：
    *   如何平衡词典的覆盖面与插件的大小？（建议只收录最常用的 5000-10000 个外来语词条）。
    *   JSON 查找在大量文本下是否会卡顿？（需要使用高效的索引结构，如 Map 或 Bloom Filter）。
2.  **多国语言歧义**：
    *   同一个片假名可能对应多个语言的原词（如「アルバイト」源自德语 Arbeit，「パン」源自葡萄牙语 pão）。
    *   离线词典需要能够记录源语言信息。
3.  **UI 冲突**：
    *   目前插件已经在显示振假名，如果再叠加外语原词，页面可能会变得非常拥挤，需要精心设计 CSS 样式（例如使用不同颜色的下划线或悬浮窗）。
4.  **形态学分析误差**：
    *   `Kuromoji` 有时会把长句子切分错误，导致无法正确匹配词典条目。
